# Spotify User Lifecycle Predictor: Comprehensive Project Documentation

This document provides a detailed breakdown of the Spotify Churn Prediction project. It is designed for beginners to understand not just *what* happens, but *how* and *why* it happens.

---

## 🌟 1. Project Overview & Objective
The goal of this project is to predict whether a Spotify user will "churn" (stop using the service) within 30 days. Instead of just looking at who already left, we build a system that identifies "leading indicators"—behaviors that happen *before* someone leaves—so the business can take action early.

---

## 🛠 2. The Tech Stack (What tools are we using?)

| Library | What it does in "Plain English" |
| :--- | :--- |
| **Pandas** | The "Excel of Python." It handles all our data tables (DataFrames). |
| **NumPy** | Handles complex math and numbers behind the scenes. |
| **Matplotlib / Seaborn** | Our "Artist tools." They turn boring numbers into beautiful charts and graphs. |
| **DuckDB** | A super-fast SQL engine that runs inside Python. We use it to write professional database queries on our CSV files. |
| **XGBoost** | The "Brain." It's a powerful Machine Learning algorithm that learns patterns from old data to predict future behavior. |
| **Scikit-Learn** | A toolkit for "teaching" the brain. It helps us split data into training/testing sets and evaluate how smart our model is. |
| **Boto3** | A "bridge" to AWS (Amazon Web Services). It's used to pull data from cloud storage. |

---

## 🔄 3. The Project Flow (How things connect)
1. **Discovery (`01_retention_analysis`)**: We look at the data to find out when most people quit (The "D7" or 7-day mark).
2. **Feature Engineering (`02_sql_feature_engineering`)**: We use SQL to calculate "smart" features, like a user's 7-day rolling average of listening time.
3. **Prediction (`03_xgboost_predictor`)**: we feed the smart features into the XGBoost model to train it.
4. **Insights**: We look at "Feature Importance" to see which behaviors (like skipping too many songs) most strongly predict churn.

---

## 📂 4. Detailed File & Code Breakdown

### File 1: `01_retention_analysis.ipynb` (The Detective Phase)
This notebook is about understanding the user's journey.
*   **Stage 1 & 2 (Setup & Ingestion)**: We load the libraries and the `spotify_churn_dataset.csv`. We also include a code pattern for `boto3` to show how this would work if the data were stored on Amazon S3.
*   **Stage 3 (Retention Curves)**: We calculate the probability of a user staying based on how many days it's been since their last login.
    *   *Key Concept*: **D7 Critical Point**. The graph shows a massive drop at 7 days. This tells us the first week is the most important time to keep a user engaged.
*   **Stage 4 (Behavioral Signals)**: We use "KDE Plots" and "Violin Plots" to see if churned users behave differently.
    *   *Finding*: Churned users usually have fewer `avg_daily_minutes` and higher `skips_per_day`.

### File 2: `02_sql_feature_engineering.ipynb` (The Prep Phase)
Data in its raw form isn't always enough. We create new "Engineered Features" using SQL.
*   **DuckDB Initialization**: We register our Pandas table as a SQL table so we can talk to it using SQL language.
*   **The SQL Query (The Heart of the Project)**:
    *   `PERCENT_RANK()`: This tells us if a user is in the top 10% or bottom 10% of listeners.
    *   `negative_feedback_density`: We divide skips by minutes. A high score means the user is frustrated (skipping a lot but not listening much).
    *   `rolling_7d_intensity_avg`: This looks at the last 7 rows of data to see if a user's activity is trending up or down. This is a "Window Function."
*   **Materialization**: We save this new, "smarter" dataset as `engineered_lifecycle_features.csv`.

### File 3: `03_xgboost_predictor.ipynb` (The AI Phase)
Now we build the predictive model.
*   **Categorical Encoding**: Computers don't understand words like "Premium" or "US". We use `get_dummies` (One-Hot Encoding) to turn these into 1s and 0s.
*   **Sanitization**: XGBoost is picky about column names. We use "Regular Expressions" (`re`) to remove symbols like `[` or `]` from our column headers.
*   **Data Splitting**: We split our data into 80% for "Learning" (Training) and 20% for "Testing" (Validation).
*   **Training the Model**: We use `XGBClassifier`. We set the `learning_rate` to 0.06 to make sure it learns carefully and doesn't "overfit" (memorize the data instead of learning patterns).
*   **Evaluation**:
    *   **Confusion Matrix**: A 2x2 grid showing how many times the AI was right vs. wrong.
    *   **Feature Importance (Gain)**: The final chart. It shows that our SQL-engineered features (like `intensity_percentile`) are actually the most important things the model uses to make its decisions!

---

## 📈 5. Summary for a Beginner
In short: We took raw Spotify logs, used math and SQL to create "health scores" for each user, and then taught a computer model to recognize the "fingerprint" of a user who is about to quit. The project proves that **how** you listen (skips vs. minutes) is more important than **where** you live (country) for predicting churn.
